# AI 辅助测试用例生成 Skill 提示词

## 一、技能概述

**技能名称**: AI 辅助测试用例生成  
**适用场景**: 数仓测试用例自动化生成，根据 Mapping 文档、表结构 DDL、需求文档 (RS/TS) 生成符合规范的测试用例及验证 SQL

---

## 二、输入规范

### 2.1 必需输入

| 参数名 | 类型 | 说明 | 格式要求 |
|--------|------|------|----------|
| `mapping` | 文件 | Mapping 映射文档（Markdown 表格格式） | 包含表级 mapping 和字段级 mapping 两个表格 |
| `TS` | 文件 | 表结构设计文档 | 包含功能说明、物理模型设计、变更记录 |
| `w3_id` | 文本 | 用户工号（接收任务完成通知） | 格式：字母 + 数字，如 q00797588 |

### 2.2 可选输入

| 参数名 | 类型 | 说明 |
|--------|------|------|
| `RS` | 文件 | 需求规格文档 | 用于提取测试要点 |
| `DDL` | 文本 | 建表语句 | 用于表结构校验 |
| `sys.query` | 文本 | 用户当前输入/指令 |

### 2.3 对话上下文变量

| 变量名 | 说明 | 用途 |
|--------|------|------|
| `conversation.user_w3_id` | 用户 w3 账号 | 存储接收通知的工号 |
| `conversation.DDL` | DDL 建表语句 | 表结构参考 |
| `conversation.RS` | 提取的测试要点 | RS 文档分析结果 |
| `conversation.mapping_table1` | 实体级 mapping | 表级映射关系 |
| `conversation.mapping_table2` | 属性级 mapping | 字段级映射关系 |
| `conversation.test_case` | 生成的测试用例描述 | LLM 生成的 JSON 用例 |
| `conversation.test_case_naotu` | 测试用例脑图 | Mermaid 格式的思维导图 |

## 三、执行流程

### 3.1 主流程概览

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           用户输入 (start)                                   │
└─────────────────────────────────────────────────────────────────────────────┘
                                      │
                                      ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│  步骤 1: 意图识别分类 (LLM)                                                  │
│  - 输入：Mapping 内容 + RS 内容 + TS 内容 + 用户查询                           │
│  - 输出：JSON {class_reason, class_type: 1/2/3/4}                           │
└─────────────────────────────────────────────────────────────────────────────┘
                                      │
                    ┌─────────────────┼─────────────────┐
                    │                 │                 │
                    ▼                 ▼                 ▼
            ┌───────────┐    ┌───────────┐     ┌───────────┐
            │ class=1   │    │ class=2   │     │ class=3   │
            │ 闲聊/引导 │    │ 确认脑图  │     │ 初次生成  │
            │ 非任务式  │    │ 生成用例  │     │ 开始任务  │
            └───────────┘    └───────────┘     └───────────┘
```

### 3.2 详细执行步骤

#### 步骤 1: 文档解析与预处理

| 子步骤 | 节点类型 | 处理内容 |
|--------|----------|----------|
| 1.1 | document-extractor | 解析 mapping 文档，提取文本内容 |
| 1.2 | code (Python) | 格式化 mapping：用正则 `split_markdown_tables()` 分离表级/字段级表格 |
| 1.3 | if-else | 判断 RS 文档是否存在 |
| 1.4 | document-extractor | 解析 RS 文档（如存在） |
| 1.5 | code (Python) | 提取 RS 中"测试要点"章节内容 |
| 1.6 | document-extractor | 解析 TS 文档 |
| 1.7 | LLM | TS 文档要素提取：schema、表名、主键、分布方式、变更内容 |

#### 步骤 2: 意图识别与路由

**LLM 意图分类器 Prompt 核心逻辑**:

```
分类规则（按优先级）:
- Class 1 (闲聊/引导): 材料缺失、咨询问题、打招呼
- Class 2 (确认执行): 用户认可脑图，要求生成详细用例
- Class 3 (初次生成): 用户提供材料，请求开始生成测试
- Class 4 (其他): 无法识别的意图

输出格式：{"class_reason": "归类依据", "class_type": 1/2/3/4}
```

#### 步骤 3: 测试要点提取（仅 class=3 时执行）

| 分支 | 条件 | 处理方式 |
|------|------|----------|
| 成功分支 | 代码成功提取测试要点 | 直接使用代码提取结果 |
| 兜底分支 | 代码提取为空 | 调用 LLM 兜底提取测试要点 |

**LLM 兜底提取 Prompt**:
```
角色：文档分析师
任务：
1. 考虑本次需求修改点，针对性生成测试要点
2. 提取"测试要点"章节内容（根据修改内容筛选，不全部照搬）
3. 排除 DQ（数据质量检查）类内容
```

#### 步骤 4: 知识检索（可选）

- **检索库**: 测试用例设计规范知识库
- **查询词**: 用户输入 `sys.query`
- **配置**: top_k=3, 启用重排序 (gte-rerank)

#### 步骤 5: 测试用例脑图生成（Agent）

**Agent 配置**:
- **模型**: privacy_DeepSeek-v3.1-w8a8-Instruct-Long
- **工具**: 代码解释器、数据库查询 SQL
- **最大迭代**: 10 次

**核心 Prompt 指令**:
```markdown
# Role: 资深数据仓库测试专家

# Task: 根据 Mapping、表属性、RS 测试要点规划测试用例脑图

# Context:
1. Mapping 逻辑（表级 + 字段级）
2. 表属性及变更内容
3. 测试用例设计规范（知识检索结果）
4. 必须包含的测试用例（RS 提取）
5. 常见测试用例示例

# Requirements:
1. 禁止幻觉产生不存在的事实
2. 区分优化类/新增类需求
3. 输出 mermaid 格式脑图，包含：
   - 用户指定的测试要点（必选）
   - 基础一致性（可选）
   - 字段级逻辑（可选）
   - 数据质量（可选）
   - 性能相关（可选）
   - 业务相关（可选）
4. L3 层级测试用例禁止模糊表达，需定义具体指标
5. 最终测试用例不超过 20 个

# Output Format:
graph LR
    root("测试用例脑图")
    L1_1("用户指定的测试要点")
    L3_1_1("[类型][操作][对象]_预期结果")
```

#### 步骤 6: 用户确认脑图

- **Answer 节点**: 向用户展示脑图并请求确认
- **用户回复**: "确认脑图正确，生成测试用例"

#### 步骤 7: 测试用例详细生成（迭代处理）

**迭代结构**:
```
┌────────────────────────────────────────────────────┐
│  迭代 1768483057445 (遍历每个测试用例 item)         │
├────────────────────────────────────────────────────┤
│  7.1 判断是否需要生成 SQL (code)                    │
│  7.2 分支：需要写 SQL                              │
│      ├── 7.2.1 提取 mapping 关键信息 (LLM)          │
│      ├── 7.2.2 查询知识库 few-shot (Agent)         │
│      ├── 7.2.3 生成测试用例 SQL (Agent)            │
│      ├── 7.2.4 执行 SQL 验证 (Tool)                │
│      └── 7.2.5 合并结果 (code)                      │
│  7.3 分支：无需写 SQL                              │
│      └── 直接输出 Agent 处理过程                    │
└────────────────────────────────────────────────────┘
```

**迭代内子步骤详解**:

| 步骤 | 节点 | 功能 |
|------|------|------|
| 7.1 | code | 判断 `need_generate_sql` 字段 |
| 7.2.1 | LLM | 从 mapping 中提取与当前测试用例相关的关键信息（最小够用原则） |
| 7.2.2 | Agent | 查询知识库获取相似用例 SQL few-shot |
| 7.2.3 | Agent | 生成 GAUSS DB 验证 SQL（支持 ReAct 迭代修正） |
| 7.2.4 | Tool | 执行 SQL 到数据库，返回结果 |
| 7.2.5 | code | 合并 SQL、执行结果、Agent 思考过程到用例 |

**Agent-生成 SQL 核心 Prompt**:
```markdown
# Role: SQL 专家

# Workflow:
Step 1 策略分析 - 阅读参考范例，确定 SQL 风格
Step 2 编写初版 - 根据 Mapping 和 DDL 编写 SQL
  - 表名严格遵循 schema.table_name 格式
  - 包含 CASE WHEN 输出'PASS'/'FAIL'
Step 3 执行与诊断:
  - 情况 A: 报错 → 重新阅读 DDL/mapping，修正后重试
  - 情况 B: 成功 → 无论 PASS/FAIL 都停止，进入 Step 4
Step 4 最终输出：{"sql": "最终验证 SQL"}
```

#### 步骤 8: 结果聚合与导出

| 步骤 | 节点类型 | 功能 |
|------|----------|------|
| 8.1 | code | 格式化测试用例为 JSON 列表 |
| 8.2 | HTTP Request | 调用 Excel 生成服务 `http://10.31.169.36:9002/generate_excel` |
| 8.3 | Tool | 发送完成通知到用户 w3 账号 |
| 8.4 | Answer | 向用户展示最终结果 |

## 四、输出规则

### 4.1 输出格式

**阶段 1 - 脑图输出**:
```markdown
## 测试要点脑图

graph LR
    root("表 A, 表 B<br/>测试用例脑图")
    L1_1("用户指定的测试要点")
    L1_2("基础一致性")
    L3_1_1("[IT 用例][表视图检查][schema.table]_预期结果")
```

**阶段 2 - 详细用例输出 (JSON)**:
```json
[
  {
    "case_name": "目标表主键不重复校验",
    "level": "level1",
    "pre_condition": "源表任务已完成，目标表任务已完成",
    "need_generate_sql": true,
    "eval_step_descri": "测试步骤描述",
    "expected_result": "查询结果为空，无重复主键",
    "tags": "IT 用例表视图"
  }
]
```

**阶段 3 - Excel 文件**:
- HTTP POST 到生成服务
- Body: `{"test_cases": <格式化后的用例列表>}`

**阶段 4 - 完成通知**:
- 通过 welink/Outlook 发送消息到 `user_w3_id`
- 内容：生成的 Excel 文件内容

### 4.2 命名规范

测试用例命名格式：
```
[IT 用例/业务用例/配置调度][条件/操作][对象英文名]_预期结果
```

示例：
- `IT 用例表视图检查 schema.table_name_验证主键唯一性`
- `业务用例业务逻辑检查 schema.table_name_验证金额 RMB 与 USD 倍数在 6~8`

### 4.3 输出约束

| 约束项 | 说明 |
|--------|------|
| 数量限制 | 最终测试用例不超过 20 个 |
| L3 描述 | 禁止模糊表达，必须定义具体指标（如"成功率 100%"而非"验证正确"） |
| 字段引用 | 涉及验证的字段/表/视图必须明确标出，不可用"等"省略 |
| Schema 要求 | 对象命名需带 schema（从 mapping 或 DDL 获取） |
| SQL 规范 | GAUSS DB 关键词需小写，表名/字段名需小写或用 lower/upper 转换 |

## 七、关键代码片段

### 7.1 Mapping 表格分割 (Python)
```python
def split_markdown_tables(s):
    pattern = re.compile(r'(?:\|.\n)+')
    tables = pattern.findall(s)
    tables = [table.strip() for table in tables]
    tables = [table for table in tables if table.count('|') > 4]
    return tables
```

### 7.2 测试用例 JSON 提取 (Python)
```python
def extract_json_from_text(text):
    match = re.search(r'\[(.)\]', text, re.DOTALL)
    if match:
        json_str = '[' + match.group(1) + ']'
        return json.loads(json_str)
```

## 八、异常处理策略

| 异常场景 | 处理方式 |
|----------|----------|
| RS 文档不存在 | 赋值为"不涉及"，继续流程 |
| 测试要点提取失败 | LLM 兜底提取 |
| SQL 执行报错 | Agent 迭代修正（最多 3 次） |
| 知识库检索为空 | 输出"未找到可参考的业务逻辑或 SQL" |
| Excel 生成失败 | HTTP 请求重试 3 次 |

## 九、使用示例

### 场景 1: 用户首次上传材料
```
用户输入：上传 mapping.md 和 TS.docx 文件，并输入"帮我生成测试用例"
系统执行：
1. 解析 mapping 和 TS 文档
2. 意图识别为 class=3（初次生成）
3. 生成测试用例脑图
4. 等待用户确认
```

### 场景 2: 用户修改脑图
```
用户输入："脑图中增加一个分区验证的测试点"
系统执行：
1. 意图识别为 class=1（修正脑图）
2. Agent 重新规划脑图
3. 展示修改后的脑图
```

### 场景 3: 用户确认生成
```
用户输入："确认脑图正确，生成测试用例"
系统执行：
1. 意图识别为 class=2（确认执行）
2. 迭代处理每个测试用例
3. 生成 SQL 并执行验证
4. 导出 Excel 并发送通知
```

## 十、注意事项

1. **数据依赖顺序**: 必须先有 mapping 和 TS 文档才能开始生成任务
2. **迭代并行度**: 迭代处理测试用例时 `parallel_nums=3`
3. **SQL 执行安全**: 仅允许 SELECT 查询，禁止写操作
4. **会话状态管理**: 脑图生成后需等待用户确认才能进入详细用例生成
5. **命名唯一性**: 测试用例名称需唯一，避免重复
